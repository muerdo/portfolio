The False Sense of Security: The Common 15-Minute Mistake That Downs Supabase Apps

TOPIC: CLOUD SEC // READ: 8 MIN // CRITICAL
Executive Summary: 80% of Supabase implementations I audit have the same flaw: exposing Postgres Views to the public API without RLS (Row Level Security). This essentially creates a "Select * From Users" button for any anonymous attacker.

The Incident: "I Just Lost My Entire Database"

Last week, I received a frantic call from a founder. Their e-commerce platform, built with Next.js and Supabase, had been wiped. Even worse: the attacker sent an email listing the name, email, and address of every single customer.

They weren't using simple passwords. They had 2FA. They had protected their API Keys. So what happened?

GET /rest/v1/admin_users_view?select=*
HTTP/1.1 200 OK

An unsecured View was exposing the administrative data.

Technical Analysis of the Vulnerability

Supabase is magical because it exposes your database directly as a REST API. To protect data, you use RLS (Row Level Security).

However, many developers create SQL Views to simplify complex queries used in the frontend.

THE TRAP: By default, in Postgres, Views execute with the permissions of the USER WHO CREATED THE VIEW (usually the admin), NOT the user running the query, unless configured otherwise. Furthermore, Views in Supabase often BYPASS RLS if not explicitly configured with `security_invoker`.

The Misconfiguration

Common code found in migrations:

-- VULNERABLE VIEW
CREATE VIEW public.analytics_dashboard AS
  SELECT 
    u.email, 
    p.total_spent 
  FROM auth.users u
  JOIN profiles p ON u.id = p.id;

The Supabase API (PostgREST) treats this view as a queryable endpoint. If you haven't explicitly restricted access to this View in the API settings or via GRANTs, it is public.

Notice to 'AI-Native' Developers

With the rise of Cursor, Windsurf, and other AI coding assistants, I have seen an explosion of this vulnerability. Why?

LLMs (Large Language Models) tend to write standard SQL code, i.e., `CREATE VIEW...`. They rarely check documentation for specific frameworks like Supabase unless explicitly prompted.

When you ask an AI to "Create a view combining users and orders", it generates standard SQL. It effectively creates a backdoor because it forgets to add with (security_invoker=true).

Recommendation: Always explicitly prompt your AI: "Create a secure RLS-compliant view for Supabase".

The End of the False Sense of Security

Many frontend devs believe Supabase handles security automatically. Wrong. Supabase is a PostgreSQL exposed via API. The database security rules are your responsibility.

The 15-Minute Solution

If you have a Supabase project in production, stop everything and check these points:

Action Command / Config Why?
Secure Views with (security_invoker=true) Forces the View to use the permissions of the current request user (RLS), not the creator.
Revoke Public Access REVOKE ALL ON logic_function FROM PUBLIC; Prevents anonymous users from executing internal database functions.
Service Role Key Never use `service_role` key in frontend env vars. This key bypasses ALL RLS. It is for backend admin scripts only.

Safe Pattern for Views:
create view public.safe_profile as 
select * from profiles 
with (security_invoker=true);
